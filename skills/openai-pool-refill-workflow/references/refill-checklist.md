# Refill Troubleshooting Checklist

按问题类型使用对应清单。不要跳步。

## 场景 1：没有触发自动补号

按顺序检查：

1. `auto_register` 是否开启
2. 固定代理是否配置，或 `proxy_pool_enabled` 是否开启
3. `_state.status` 是否为 `idle`
4. CPA gap 是否大于 0
5. Sub2Api gap 是否大于 0
6. `auto_sync` 是否开启（影响 Sub2Api gap 是否参与）
7. `upload_mode` 是否导致总 gap 小于预期
8. `_state.start_task(...)` 是否抛出运行态冲突

输出时要明确说明：
- 是哪一个前置条件没满足
- 还是 gap 算出来本来就是 0

## 场景 2：补号数量不对

按顺序检查：

1. `min_candidates` / `sub2api_min_candidates`
2. 池状态接口返回的 candidates 实际值
3. `calculate_gap()` 的输入值是否正确
4. `snapshot` / `decoupled` 的 gap 合成规则
5. 是否只配置了单平台补充
6. 是否把 worker 数误当成补号目标数

输出时要明确说明：
- 目标数错在阈值、候选数、策略合成，还是任务启动参数

## 场景 3：补号任务启动了，但池数量没涨

按顺序检查：

1. 注册是否真正成功生成 token
2. token 是否写入 `data/tokens/`
3. 平台上传函数是否成功
4. 平台统计接口是否把新增账号计入 candidates
5. 前端状态是否只是没刷新

输出时要明确说明：
- 卡在注册、上传、统计还是前端展示

## 场景 4：维护后没补号

按顺序检查：

1. 维护线程是否真的执行
2. 维护完成后是否调用 `_try_auto_register()`
3. 调用时的配置快照是否正确
4. gap 是否在维护后已经变成 0
5. `_state.status` 是否阻止再次启动

输出时要明确说明：
- 是维护没跑到补号分支
- 还是补号分支运行了但被条件拦住

## 场景 5：前端显示异常，像是没补号

按顺序检查：

1. 后端接口返回值
2. SSE 或状态广播内容
3. `static/app.js` 的刷新函数和状态文案
4. 刷新动作是否在操作成功后触发

输出时要明确说明：
- 是后端状态不对，还是前端消费/渲染不对

## 统一输出模板

分析完后，尽量按这个结构回复：

1. **问题归类**：属于哪种补号问题
2. **关键链路**：这次走到哪些函数/文件
3. **根因**：真正卡住的条件或逻辑
4. **影响范围**：是否影响 CPA、Sub2Api、双平台策略或前端展示
5. **修改点**：准备改哪几个文件，为什么
